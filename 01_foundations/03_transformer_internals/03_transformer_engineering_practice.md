# 03 Transformer工程化应用实践

## 一、对话 Agent 的两种工作模式

结合 RNN 编解码范式，可将 Transformer 在 Agent 中的落地理解为两种模式。

### 模式1：纯闲聊生成（编码器 + 解码器）

流程：

1. 用户输入 -> 编码器得到上下文语义向量
2. 解码器以 `<bos>` 与历史已生成 token 为输入
3. 掩码自注意力（不看未来）+ 交叉注意力（看编码器输出）
4. FFN -> 线性层 -> Softmax，得到下一 token 概率
5. 自回归迭代直到 `<eos>`

与双层 RNN 编解码对比：

- 相同点：都属于编码-解码范式
- 不同点：
  1. Transformer 编码并行，RNN 串行
  2. Transformer 依赖注意力关联全局，RNN 更依赖隐状态链传递
  3. 长上下文下 Transformer 通常更稳

### 模式2：工具/API调用（仅编码器）

流程：

1. 用户输入 -> 编码器 -> 句向量（`[CLS]` 或 Mean Pooling）
2. 与 API 技能向量库做相似度匹配（余弦相似度）
3. 阈值判断：
   - 高于阈值：触发对应工具/API
   - 低于阈值：回退到模式1生成回复
4. 参数抽取（地点、日期、实体等）并执行调用
5. 返回结果后可选用生成模型润色输出

## 二、编码器做意图分类/API匹配的核心逻辑

1. 预先构建技能描述及其向量索引
2. 在线将用户输入编码为句向量
3. 计算相似度并排序，取 Top-1 或 Top-k
4. 结合阈值与规则完成路由决策

这类方案可在无额外监督分类器时快速搭建“语义路由”。

## 三、解码器生成文本的核心逻辑

1. 自回归：每次生成一个 token
2. 因果掩码：禁止看到未来位置
3. 交叉注意力：持续对齐输入语义
4. Softmax：从词表概率中采样/贪心/束搜索得到输出

## 四、两种模式切换建议

1. 统一先做编码器语义编码
2. 先走工具匹配（高置信度任务优先）
3. 命中工具则调用，不命中则生成式回答
4. 对关键任务增加安全校验与回退策略

## 五、实践总结

1. 闲聊/翻译：通常需要编码器 + 解码器
2. 意图识别/API路由/RAG检索：编码器句向量即可覆盖多数需求
3. 相比传统 RNN 管线，Transformer 在并行性与长上下文处理上更适合现代 Agent 系统
4. 句向量可根据场景选 `[CLS]` 或 Mean Pooling：实时路由偏前者，语义检索偏后者
